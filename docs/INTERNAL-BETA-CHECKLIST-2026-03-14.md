# 内部试用清单（2026-03-14）

## 1. 目标

本轮目标不是验证“AI 是否能全自动替代客服”，而是验证下面 4 件事是否成立：

1. 客服能更快拿到可复制的回复建议。
2. 高风险问题在证据不足时会保守兜底，不会乱答。
3. 新导入的产品知识库和 Excel Q&A 能被全局问答正确命中。
4. 问题反馈可以被稳定记录，便于后续修正。

## 2. 本轮内测边界

- 当前环境：本地 FastGPT 社区版
- 当前部署：本机后端 + 本地 FastGPT Docker + `kb-server.local`
- 本轮先不改 FastGPT 默认密码、MinIO 密码和示例密钥
- 因为仍是本地测试环境，所以只允许本机或可信内网小范围试用，不做公网暴露

## 3. 本轮测试范围

### 3.1 进入测试范围的知识库

- `tysl-local-kb-rules`
- `tysl-local-kb-refund`
- `tysl-local-kb-complaint`
- `tysl-local-kb-activate`
- `tysl-local-kb-logistics`
- `tysl-local-kb-acct-sec`
- `tysl-local-kb-baichuan`
- `tysl-local-kb-ebo`
- `tysl-local-kb-device-shop`
- `tysl-local-kb-home-ai`
- `tysl-local-kb-b2b-ict`

### 3.2 暂不作为内测通过标准的范围

- `internal-support` 内部支持库
- 新增但尚未验证的新文档
- 单文档问答的“严格硬隔离”能力

说明：
- 单文档问答已经可用，但社区版工作流暂不支持按 `collectionId/sourceName` 做搜索前硬过滤。
- 所以内测重点应放在全局问答和客服操作效率，不把“绝不串文档”作为当前通过标准。

## 4. 谁来测

建议参与角色：

- `1` 位客服班长：判断口径是否可落地
- `2-4` 位一线客服：重点体验是否省时间、是否好复制
- `1` 位产品或运营：记录知识缺口和错误归因

## 5. 测什么问题

### 5.1 必测高风险题

每类至少测 `3` 条，优先使用真实工单改写后的问法：

1. 退款
2. 投诉升级
3. 账号安全
4. 激活绑定
5. 物流异常

建议问法示例：

- 用户要求退款，客服应该如何回复并说明处理时效？
- 用户投诉客服态度差，并要求主管回电，应该怎么安抚和升级？
- 用户反馈账号疑似被盗，需要修改绑定信息，客服怎么处理？
- 用户反馈设备一直无法绑定，应该怎么一步步排查？
- 用户说物流显示已签收但本人没有收到，客服该怎么回复？

### 5.2 必测产品题

每个产品库至少测 `2-3` 条：

- `视联百川客户端（baichuan）`
- `EBO-SE 移动机器人（ebo）`
- `商城设备类产品（device-shop）`
- `天翼看家 AI 产品（home-ai）`
- `政企版 AI 产品（b2b-ict）`

建议覆盖：

1. 产品功能说明
2. 开通/绑定/登录/配网
3. 常见失败场景
4. 适用范围或限制条件

### 5.3 必测负向题

这类题目不是为了“答出来”，而是为了验证系统是否会保守兜底：

1. 故意问错产品线的问题
2. 故意只给半句话，不给关键信息
3. 故意问文档外结论
4. 故意问需要人工审批或赔付承诺的问题

判定标准：

- 允许追问
- 允许转人工
- 不允许瞎承诺
- 不允许无证据给结论

## 6. 怎么测

### 6.1 客服试用动作

每位测试者按下面顺序体验：

1. 打开全局问答
2. 输入真实或半真实客服问题
3. 判断是否能直接复制使用
4. 查看是否有引用
5. 若答案有问题，立即记录

如果要体验单文档问答，再额外做：

1. 进入某篇知识文档
2. 问一条文档内问题
3. 问一条文档外问题
4. 判断是否会保守兜底

### 6.2 推荐测试节奏

- 每次试用控制在 `30-45` 分钟
- 每人每天提交 `5-10` 条有效问题即可
- 第一天重点看“会不会乱答”
- 第二天开始再看“能不能省时间”

## 7. 怎么记问题

建议统一记到一个共享表格，至少保留下面字段：

| 字段 | 说明 |
| --- | --- |
| 日期 | 例如 `2026-03-14` |
| 测试人 | 班长/客服/产品 |
| 场景分类 | 退款/投诉/账号/激活/物流/产品问答 |
| 问题原文 | 客服真实输入或改写输入 |
| 命中的知识库 | 如可见则记录；不可见可留空 |
| 是否有引用 | 有 / 无 |
| 是否可直接复制 | 可直接用 / 需小改 / 不可用 |
| 问题类型 | 乱答 / 漏答 / 串产品 / 无引用 / 追问不足 / 文案太长 / 其他 |
| 正确口径 | 客服认为正确的回复或补充说明 |
| 严重程度 | Blocker / High / Medium / Low |
| 截图或链接 | 可选 |

### 7.1 最低记录要求

如果现场很忙，最少也要记这 `5` 项：

1. 问题原文
2. 错误类型
3. 正确口径
4. 是否有引用
5. 严重程度

## 8. 哪些算 blocker

下面这些直接算 `blocker`，当天必须拦住继续放量：

1. 高风险问题无证据却给出明确结论
2. 串产品线，且回复看起来像正确答案
3. 暴露敏感信息、内部联系方式、私网信息或内部口径
4. 服务不可用、页面无法加载、连续报错
5. 引用明显错误，且会误导客服执行错误动作
6. 对退款、投诉升级、账号安全、赔付承诺给出错误口径

## 9. 哪些不算 blocker

下面这些先记为 `High` 或 `Medium`，不一定当天阻断：

1. 文案偏长，但结论正确
2. 需要客服手工精简后才能发给用户
3. 回答保守，倾向人工复核
4. 部分产品问题只答到一半，但没有乱答
5. 图片没有完全显示，但文字回答仍正确

## 10. 每天怎么收口

建议每天结束时只做 3 件事：

1. 汇总 blocker 数量和具体样本
2. 汇总高频错误类型前 `5` 名
3. 明确第二天只修最影响体验的 `1-3` 件事

推荐分类维度：

- 知识缺口
- 检索不到
- 串产品
- 高风险兜底不够硬
- 文案可复制性差

## 11. 本轮通过标准

如果满足下面条件，可以继续扩大内测范围：

1. 连续 `2` 天没有新增 blocker
2. 高风险题的错误口径为 `0`
3. 主产品题大部分都能命中引用
4. 一线客服主观反馈为“有帮助，不比人工搜索更慢”

## 12. 本轮暂定人工兜底规则

内测期间统一口径：

1. 无引用答案不作为最终执行依据
2. 涉及退款、投诉升级、账号安全、赔付、内部支持时，以人工复核为准
3. 单文档回答若明显超出当前文档范围，以人工判断为准

## 13. 建议下一步优化

内测记录的后续去向建议分成两类：

1. `知识问题`
   - 缺知识
   - 旧知识版本过期
   - Q&A 拆分不够细
2. `系统问题`
   - 检索错
   - 高风险问题未硬兜底
   - 文案不利于复制
   - 需要增加人工纠错入口

本轮内测结束后，建议优先推进：

1. 单文档无有效引用时的后端硬兜底
2. 客服人工反馈标注入口
3. 新文档先入影子库，再自动抽取客服可用层知识
