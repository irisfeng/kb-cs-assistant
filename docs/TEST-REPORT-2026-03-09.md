# 测试报告与模型切换评估

日期：2026-03-09
项目：客服知识助手 `kb-cs-assistant`

## 1. 结论摘要

当前系统的主链路“知识提报 -> 审核 -> 发布 -> 文档预览”可继续作为可用基础能力保留。

但问答链路存在明显分化：

- 全局问答可返回内容，但当前更接近“保守兜底模板”，实测未稳定产出引用。
- 单文档问答当前存在明显退化，接口实测直接返回通用欢迎语，未形成有效文档内问答。

关于模型切换：

- 全局问答从 `gpt4mini` 切到 DeepSeek V3，从仓库代码结构看“可以做”，且大概率不需要改业务代码。
- 但当前仓库并没有写死 `gpt4mini`，后端只把请求发往 FastGPT app channel，真正的底层模型配置大概率在 FastGPT 侧。
- 单文档问答的对话模型也“可以切到 DeepSeek V3”，但不建议把它当成当前问题的优先修复手段，因为现阶段主要瓶颈更像是 workflow 检索与作用域约束失效，而不是纯模型能力不足。

## 2. 测试范围

本轮重点覆盖以下范围：

1. 前后端与 FastGPT 连通性
2. 前端构建与后端语法检查
3. 全局问答接口 `/api/chat`
4. 单文档问答接口 `/api/solutions/:id/chat`
5. 会话连续性与重复提问稳定性
6. 模型切换可行性与风险评估

## 6. 对当前实现的判断

### 6.1 全局问答

当前全局问答链路是可工作的，但表现更像：

- FastGPT app channel 有响应
- 后端流式转发正常
- prompt 在生效
- 检索引用链没有稳定落到前端

因此它当前适合作为“辅助分诊 / 人工前置整理”，不适合作为“高置信自动答复”。

### 6.2 单文档问答

单文档问答当前不是单纯“效果一般”，而是已经出现明显工作流退化。仓库代码层面虽做了三层补救：

1. system prompt 强限制当前文档
2. 传入 `collectionId` 和 `sourceName` 变量
3. 后端对 `quoteList` 再做一次当前文档过滤

但这三层都建立在 FastGPT workflow 本身能先正确检索并正确输出的前提上。当前实测直接返回欢迎语，说明真正失效点大概率仍在 FastGPT workflow 内部，而不是前端页面本身。

## 7. 是否沿用了老项目工作流

结论：是，单文档问答明显沿用了“旧知识库项目改造版”的实现思路，而不是为当前客服场景重新设计的一条硬隔离工作流。

依据：

1. 仓库 README 明确说明这是从原 knowledge-base workspace 克隆并改造而来。
2. 后端仍保留 legacy `draft/PPT` 代码，只是通过开关隔离，并未彻底清除。
3. 单文档问答本质上仍是“通用 FastGPT chat/completions + prompt 约束 + 变量提示 + 后过滤”的软隔离方案。
4. `docs/development-memory.md` 里也明确记录了 workflow 侧过滤仍需对齐。

这条思路在“低风险试运行”下可以勉强工作，但对当前客服场景并不完全适用，原因是：

- 客服场景需要更稳定的证据归因
- 单文档问答需要强作用域，不适合只靠 prompt 软约束
- 一旦检索节点空命中，当前 workflow 会退化成欢迎语，而不是“未找到依据，请人工复核”

## 8. 模型切换评估

### 8.1 全局问答从 gpt4mini 换成 DeepSeek V3 可以吗

结论：可以，但建议按“FastGPT 侧切换 + 回归验证”执行，不建议只凭仓库改变量名就上线。

原因：

1. 当前后端并没有把模型写死成 `gpt4mini`，而是把请求发给 `FASTGPT_GLOBAL_APP_MODEL` 对应的 FastGPT app channel。
2. 也就是说，仓库代码层面只关心 channel 名，不真正决定底层大模型。
3. 如果你们当前在 FastGPT 的 `tysl-local-app-global-v2` 里用的是 `gpt4mini`，那切到 DeepSeek V3 主要是 FastGPT app / provider 配置变更，不一定要改后端代码。

但注意：

- 这只能改善“生成表达”与“成本/速度”侧问题
- 不能单独解决“无引用”“检索空命中”“欢迎语回退”这类 workflow 问题

### 8.2 单文档问答的对话模型换成 DeepSeek V3 可以吗

结论：技术上可以，业务上不建议先做。

原因：

1. 单文档问答当前的主问题不是回答文风，而是 workflow 没有稳定返回文档内证据。
2. 在这种情况下换成 DeepSeek V3，最可能的结果是“欢迎语依旧欢迎语，只是底层模型变了”。
3. 应先修 workflow，再谈模型切换收益。

## 9. 建议的改造优先级

### P0

1. 先检查 FastGPT 的 `solution-kb-chat` workflow，确认知识检索节点是否真的按 `collectionId/sourceName` 过滤。
2. 把单文档 workflow 的兜底输出从“欢迎语”改成“当前文档未找到足够依据，请人工复核”。
3. 确保 workflow 在命中检索时保留 `quoteList`，否则前端永远拿不到引用。

### P1

1. 检查 `tysl-local-app-global-v2` 是否稳定产出 `quoteList`。
2. 为全局问答补一组固定回归集，至少覆盖退款、投诉、激活、账号安全、物流五类核心场景。
3. 修复前端构建报错与 `button` 嵌套 `button` 的结构问题。

### P2

1. 在 FastGPT 侧将全局 app 和单文档 app 分开治理。
2. 全局问答可优先评估切到 DeepSeek V3。
3. 单文档问答等 workflow 稳定后再决定是否同步切换到 DeepSeek V3。

## 10. 最终建议

当前建议不是“先全面换模型”，而是：

1. 先修单文档 workflow
2. 再做全局问答 20 条回归
3. 然后把全局问答切到 DeepSeek V3
4. 单文档问答在修复后再决定是否一并切换

这样风险最低，也最符合当前代码和实测表现。
