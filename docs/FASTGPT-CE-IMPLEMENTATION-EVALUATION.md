# FastGPT 社区版能力评估与客服知识助手落地方案（2026-03-06）

## 1. 结论先行
1. 本项目可以基于 FastGPT 社区版落地客服智能知识助手，不需要依赖商业版即可完成 MVP。
2. 文档上传建议优先走“后端代理上传”（安全、可审计），前端仅做管理入口。
3. 明确不再投入 PPT 相关能力，所有研发资源转向“知识检索 + SOP + 结构化客服回复 + 运营闭环”。

## 2. 评估基线
1. 当前评估时间：2026-03-06。
2. 参考版本基线：FastGPT 仓库 Latest Release 为 `v4.14.7.2`（2026-02-26）。
3. 评估范围：FastGPT 社区版 OpenAPI、知识库接口、商业版功能差异。

## 3. FastGPT 社区版能力矩阵（面向本项目）

| 能力 | 社区版 | 结论 | 改造建议 |
|---|---|---|---|
| 应用工作流编排（Workflow） | 支持 | 可直接用于客服问答编排 | 保留当前 `chat/completions` 工作流模式 |
| OpenAPI 对话接口 | 支持 | 可用于全局问答与场景问答 | 统一走 `/api/v1/chat/completions` |
| 知识库文本写入 | 支持 | 可满足文档入库 | 后端调用 `collection/create/text` |
| 本地文件入库（localFile） | 支持 | 可满足图片/附件入库 | 后端转传并保存 `fileId` 引用 |
| 链接入库（link） | 支持 | 可用于网页知识同步 | 增加链接抓取后台任务 |
| 知识库数据检索/列表 | 支持 | 可用于溯源与审计 | 建立“引用命中日志” |
| 多租户 SaaS、去 FastGPT 标识、SSO 等 | 商业/企业能力 | 不属于当前 MVP 必需 | 不纳入当前阶段 |
| 一些高级发布/运营/日志能力 | 部分商业版专属 | 社区版有缺口 | 在本项目后端补日志、审计与看板 |
| 知识库高级索引参数（如 `autoIndexes`） | 商业版专属 | 社区版不可依赖 | 以离线预处理、标签和规则补偿 |

说明：商业版差异以官方“Community vs Commercial”页面为准，且不同子能力随版本可能变动，需在发布前复核。

## 4. 上传方案评估（你提出的两种模式）

### 4.1 方案 A：全部后台上传（推荐）
流程：文件进入后端（管理端上传、共享目录、批处理任务）-> 后端解析 -> 后端写入 FastGPT。

优点：
1. API Key 不暴露。
2. 可做统一审计（谁上传、何时、哪些文档、是否成功）。
3. 易于接入清洗、切分、脱敏、分类标签等治理能力。

缺点：
1. 管理端交互体验需要额外建设。
2. 大文件上传时需做断点与队列。

适配结论：MVP 默认选 A。

### 4.2 方案 B：前端做上传功能（经后端中转，推荐）
流程：前端上传到本项目后端 -> 后端携带密钥写入 FastGPT。

优点：
1. 体验好，运营可自助上传。
2. 不暴露 FastGPT 密钥。

缺点：
1. 需要补上传任务状态、失败重试、进度反馈。

适配结论：作为 A 的 UI 化能力，与 A 并行，不冲突。

### 4.3 方案 C：前端直传 FastGPT（不推荐）
风险：
1. 社区版下通常需要前端持有长期 Key，泄露风险高。
2. 审计和权限边界难控。

适配结论：不采用。

## 5. 与当前代码的差距
1. 后端已有 `upload`、`chat`、`capabilities` 能力，可复用。
2. 当前仍保留大量 `draft/ppt` 路由和数据结构，需从主路径剥离，避免误导和维护成本。
3. 会话隔离、审计日志、运营指标尚不完整。
4. 部分文案存在乱码，需要统一编码和 i18n 资源化。

## 6. 落地路线（8 周）

### Phase 0（本周）：范围冻结与文档清理
1. 冻结目标：客服知识助手，不含 PPT。
2. 清理历史文档与计划，统一路线图。
3. 定义客服五大场景验收用例（退款、投诉、账号、激活、升级）。

### Phase 1（第 1-2 周）：上传与问答链路稳态
1. 完成上传双入口：
   - 后台批量导入任务（主入口）
   - 前端管理页上传（后端中转）
2. 统一文档处理流水线：解析、清洗、切分、标签、入库。
3. 统一聊天接口参数：`sessionId`、`chatId`、引用返回结构。

### Phase 2（第 3-4 周）：客服 Copilot 核心能力
1. 意图识别与场景路由（退款/投诉/账号/激活/升级）。
2. 输出结构化模板：推荐回复、处理步骤、升级建议、风险提示。
3. SOP 规则命中与引用展示。

### Phase 3（第 5-6 周）：治理与可追溯
1. SOP 与回复模板版本化（历史、对比、回滚）。
2. 上传/问答/规则变更审计日志。
3. 低置信答案回流“知识缺口池”。

### Phase 4（第 7-8 周）：运营看板与上线
1. 指标看板：TopN 问题、SOP 命中率、转人工率、失败率。
2. 业务验收 + 压测 + 发布回滚预案。

## 7. 本项目建议的优先实现顺序（2 周冲刺）
1. 删除 PPT 入口与菜单暴露，仅保留客服功能导航。
2. 补齐上传任务状态机（queued/processing/success/failed）。
3. 打通 sessionId 隔离与会话历史。
4. 增加“引用不足时的安全降级话术”。
5. 增加基础运营统计（按天问题量、未命中问题）。

## 8. 验收标准（MVP）
1. 客服 5 类核心场景均可稳定返回“引用 + 结构化建议”。
2. 文档上传成功率 > 95%，失败任务可重试。
3. 会话隔离生效，无跨会话串扰。
4. 关键操作（上传/删除/规则变更）可审计。
5. 看板可追踪未命中问题并回流补库。
