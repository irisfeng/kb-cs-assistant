# 全面测评结论（2026-03-08）

## 1. 总结

当前系统已经具备“小范围试运行”的基础，但**还不能按生产客服 Copilot 标准直接全量开放**。

本轮复测后的结论需要拆成两部分看：

1. `知识提报 / 审核 / 发布 / 预览` 主链路已经跑通，适合先发布给班长和少量运营试用。
2. `单文档问答` 和 `全局客服问答` 都还没有达到稳定生产标准，不能当作高置信自动答复能力上线。

更准确地说：

- `提报 -> 审核 -> 发布 -> 知识列表 -> 详情预览`：通过
- `单文档问答`：部分可用，但稳定性不足
- `全局问答`：明显不具备生产可靠性

## 4. 本轮代码修正

### 4.1 会话连续性修正

已修正，但只能算“部分缓解”。

问题：

1. 单文档和全局聊天在复用同一个 FastGPT `chatId` 时，后续追问会退化成泛化欢迎语。

处理：

1. 当请求里已经带有完整历史消息时，后端不再继续复用旧 `chatId`。
2. 连续追问改为依赖前端回传的显式消息历史维持上下文。

结果：

1. 确实压制了一部分由于 `chatId` 复用导致的回退。
2. 但没有根治连续追问问题，根因仍在 FastGPT 工作流检索与问答节点。

### 4.2 检索上下文增强

已新增低风险增强，但效果有限。

处理：

1. 后端会把文档标题、文件名、最近一轮对话摘要显式注入最后一问，尽量帮助 FastGPT 检索节点保留上下文。
2. 全局问答也补充了“优先依据 SOP/规则、证据不足时保守回答”的检索提示。

结果：

1. 对部分文档有改善。
2. 但对 `EBO-SE` 这类真实客服文档，稳定性仍明显不足。

## 5. 主要发现

### [P1] 全局客服问答不具备生产可靠性

这是当前最明确的阻塞项。

表现：

1. 多个验收场景没有稳定引用。
2. 在知识不足时，仍可能生成泛化建议。
3. 个别场景会出现明显缺少证据的流程、时效或补偿描述。

这意味着：

1. 全局问答不能直接作为正式客服答复依据。
2. 目前最多只能作为“辅助建议 + 人工复核”。

### [P1] 单文档问答也还不具备生产稳定性

这次复测后，单文档问答必须下调结论。

表现：

1. 同一题重复提问，结果不稳定。
2. 明确命中的问题有时能答对，有时直接回欢迎语。
3. 连续追问时，检索节点常常只盯最后一问，无法稳定继承前文上下文。

这意味着：

1. 单文档查看和预览可以先用。
2. 单文档问答如果要放出，也只能在小范围试运行中使用，不能承诺高可靠。

### [P1] FastGPT 工作流存在检索空命中与随机性

从后端抓到的原始流可以看到：

1. 同一文档、同一问题，多次请求会出现不同检索结果。
2. 某些请求里 `知识库搜索` 节点直接 `quoteList: []`。
3. 命中失败时，工作流可能直接退化成欢迎语，而不是稳定返回“未找到依据”的保守答复。

这是当前最需要在 FastGPT 侧修的地方，不是前端页面问题。

### [P2] 单文档隔离仍是软隔离

当前单文档问答主要依赖：

1. 后端 prompt 限制。
2. 后端引用过滤。

它不是 FastGPT 社区版层面的硬隔离。

### [P2] 角色权限仍未落地

现在“客服提报”和“班长审核/发布”仍主要靠流程约束，不是账号权限约束。

这意味着：

1. 适合小范围试运行。
2. 不适合直接全员开放。

## 6. FastGPT 侧需要更新什么

### 6.1 今晚不用强制改的

1. 公共知识库本身不需要立刻重建。
2. 前后端主链路不需要为了继续本机验证而回滚或重做。

### 6.2 明确建议尽快改的

1. 新建内部支持知识库，并配置 `FASTGPT_INTERNAL_DATASET_ID`。
2. 重做单文档工作流的检索策略，不要再依赖“变量 + prompt”软限制。
3. 调整全局客服工作流：证据不足时必须稳定返回保守答复，不能再输出欢迎语或臆造规则。
4. 为退款、投诉、账号安全、激活、物流异常补齐明确的 SOP 与规则知识。
