# 极简双人协作路径

## 1. 目标

当前阶段的目标不是建立完整的平台化交付体系，而是尽快把客服智能知识系统的核心功能做成“可开发、可部署、可演示、可试用”的状态。

当前适用前提：
- 只有 1 位人类开发者
- 由我持续协助代码、文档、迁移和发布方案
- 近期目标是核心功能可用，不是大规模对外生产

在这个前提下，最合适的路径是：
- 只保留 `local` 和 `demo` 两套环境
- 用最小可维护方式上云
- 用最小流程保证每次改动可回归、可部署

## 2. 推荐环境结构

### 本地环境 `local`

用途：
- 日常开发
- 功能调试
- 数据模型调整
- 接口联调

组成：
- 本地前端
- 本地后端
- 本地环境变量
- 本地测试数据

### 云上环境 `demo`

用途：
- 对外演示
- 内部试用
- 发布后验收
- 阶段性稳定版本展示

组成：
- 阿里云 ECS 上的前后端服务
- 阿里云 PostgreSQL
- 阿里云 OSS
- FastGPT 服务

结论：
- 现阶段不要再拆 `preview / staging / production`
- 先把 `demo` 作为唯一云上可用环境

## 3. 推荐技术形态

建议采用以下最简架构：

`浏览器 -> Nginx -> Node/Express -> PostgreSQL / OSS / FastGPT`

推荐原因：
- 当前前端是静态站点，适合直接由 Nginx 提供
- 当前后端是 Express，适合直接作为单服务运行
- PostgreSQL 适合承载文档、会话、SOP 和账号数据
- OSS 适合承载原始文件和导出文件
- FastGPT 继续承担知识检索和问答能力

## 4. Git 与协作规则

建议只保留两类分支：

- `main`
  - 当前可部署版本
  - 任何时候都应尽量保持可运行

- `feature/*`
  - 单个功能开发分支
  - 开发完成后合并回 `main`

推荐节奏：
1. 需求确认
2. 在 `feature/*` 上开发
3. 本地验证
4. 合并到 `main`
5. 部署到 `demo`
6. 进行云上冒烟验收

不建议现在引入：
- `develop`
- `release/*`
- 多人协作审批流
- 复杂 GitFlow

## 5. MVP 核心功能范围

前期必须做成可用的能力：
- 文档上传
- 文档入库
- 全局知识问答
- 单文档问答
- SOP 管理
- 基础会话历史
- 基础账户登录
- 基础错误日志

可以延后：
- 多租户
- 细粒度权限
- 复杂审计中心
- 运营分析大盘
- 自动化预览环境
- 灰度发布

## 6. 最小云资源清单

建议云上只先准备这些资源：

### 计算
- `1 台 ECS`
- 用途：运行 Nginx、前端静态文件、Node 后端

### 数据库
- `1 个 PostgreSQL 实例`
- 用途：存用户、文档、上传任务、会话、消息、SOP

### 文件存储
- `1 个 OSS Bucket`
- 用途：存文档原件、导出文件、后续静态附件

### 域名与证书
- `1 个域名`
- `1 套 HTTPS`

### AI 服务
- `1 套 FastGPT 可用环境`
- 与 `demo` 保持固定连接

当前阶段不必一开始就准备：
- 多台 ECS
- Kubernetes
- 独立 Redis
- 多套数据库
- 多环境对象存储

## 7. 数据库落地优先级

数据库应尽快接入，但按优先级推进。

第一批必须落库：
1. `documents`
2. `upload_jobs`
3. `chat_sessions`
4. `chat_messages`
5. `sop_items`

第二批补充：
6. `users`
7. `audit_logs`

当前不建议继续让下面这些作为主数据源：
- `db.json`
- `capabilities.json`

它们可以保留为迁移前备份，但不应再作为正式主存储。

## 8. 最小发布流程

每次发布只走这 6 步：

1. 本地完成开发
2. 本地跑最小验证
3. 合并到 `main`
4. 部署到阿里云 `demo`
5. 执行冒烟测试
6. 记录本次变更和问题

这里故意不追求复杂自动化。

原因：
- 当前维护人少
- 上线频率可控
- 手工发布更容易定位问题

## 9. 最小验证要求

每次发布到 `demo` 后，至少验证以下主链路：

1. 登录是否正常
2. 文档上传是否成功
3. 文档是否能出现在知识库列表
4. 全局问答是否正常返回
5. 单文档问答是否正常返回
6. SOP 列表是否可读写
7. 是否出现明显报错日志

只要这 7 条通过，就可以认为本次版本达到了“核心功能可用”的上线标准。

## 10. 记录与备份要求

即使是极简流程，也必须保留记录：

- 每次关键改动写入 `docs/development-memory.md`
- 关键节点打 Git commit
- 重要阶段做 Git tag
- 高风险改动前导出数据库备份

推荐 tag 方式：
- `v0.1.0-demo`
- `v0.2.0-db`
- `v0.3.0-auth`

## 11. 什么时候升级流程

当出现以下任一情况时，再升级为多环境体系：

- 开始有外部客户正式试用
- 同时在线使用人数明显增加
- 一周发布次数超过 2 次
- 数据库迁移开始频繁发生
- 每次上线都需要先验收再发布
- 出现“一个版本影响正在演示的环境”

满足这些条件后，再引入：
- `staging`
- 自动化 CI/CD
- 更完整的回滚和审批机制

## 12. 当前推荐结论

对于当前项目，最合适的现实路径是：

- 环境：`local + demo`
- 分支：`feature/* + main`
- 部署：阿里云单套 demo
- 数据：尽快切到 PostgreSQL
- 文件：尽快切到 OSS
- 发布：手工可控，小步上线

一句话总结：

先把一套云上 `demo` 做稳，确保核心功能可用，再考虑企业级交付体系。
